PROJECT 4

Data Collection and Visualization

PROJECT BRIEF

Explore modern (read: extensive and invasive) data collection by using a custom sensor suite to gather multiple types of environmental input and visualize that data in a 2D graphic format.

CONCEPT

This project aims to expand upon the core concept of environmental data collection from project 3 by combining several types of sensors and feeding the readings into a single digital space that can interpret and display all of it.

PROJECT IMPETUS

Understanding the basics of wireless communication and data feeds is valuable, but the experience taking raw data readings from a DIY hardware configuration, interpreting it in a meaingful way, and applying it to front-end design is makes this undertaking worthwhile.

PROJECT INSPIRATION

The inspirations for this project came from my previous communications systems project and the rising smarthome movement currently dominated by company ecosystems such as Apple, Google, or Amazon. I was especially drawn to the Nest thermostat, whose user interface and control mechanism I found positively delightful in comparison to any other thermostat I've used before.

PROJECT GOALS

Ultimately, I decided that the project would consist of 4 different kinds of data from multiple sensors: temperature, humidity, light and sound. My goal was to take that data and create a user friendly room environment visualizer.

SOUND SENSOR

Adding sound to the system was the most difficult part of the assembly. I tried three different sensors over the course of three weeks until I finally found one that was compatible with Arduino and the rest of my hardware in addition to being sensitive enough to actually provide useful readings.

I2S MEMS Breakout

This microphone simply didn't work in my version of Arduino and the documentation offered no answers as to why. Even after installing the correct libraries and testing the example code, nothing was ever outputted into the serial monitor.

Piezo Sensors

Piezo sensors work by detecting vibrations in solid material such as glass or certain types of wood as a result of sound waves. Frankly, it didn't provide any useful readings as it wasn't sensitive enough and was rather finicky in what it detected.

Grove Sound Sensor

Ultimately this was the sensor chosen. It's a simple analog sound sensor that picks up noise close to the microphone very well and did a passable job at detecting ambient noise.

PROTOTYPE

SENSOR READINGS

Over the course of about 24 hours, I collected data from a bedroom and uploaded the readings every 10 seconds to adafruit.io: an online data feed service made for their wi-fi enabled microcontrollers. Unfortunately Adafruit throttles feeds as a defense mechanism against cyber attacks, therefore the 10 second interval is there to ensure against a false DOS attack alarm which is against the terms of service. In a purpose-built system, this pause would be unnecessary.

GUI OUTPUT

I then took those readings from adafruit.io and put them into a p5.js program I coded out to visualize the data. There are several different ways to handle the data feeds: I could take every single data point (or a sample size) and display the average readings or I could set it up so that it's a live data reader that uses only the last data point.

Using color language, plain text, subjective captions, and motion graphics, I hoped that the UI would not only serve as a informational dashboard but also convey the feeling that the data means. This particular design is set in a 9:16 ratio common to many smartphones.

ITERATIONS

However, there were several flaws with the first design. The noise readings didn't fit well with the rest of the data and while the perlin waves were an interesting visual component, it wasn't particularly understandable from a glance and didn't really provide anything beyond a rough measurement which isn't terribly useful. Not all data should be grouped together. Thus, I created another GUI that only contains relevant information and fit it to a circular interface similar to the Nest thermostat.

PROJECT OUTCOME

While there are certainly limitations to this system, primarily due to Adafruit's data throttling, I feel like I achieved the core goal of this project which was to interpret and visually display sensor data. It didn't really matter that I used temperature, humidity, light, and sound. It could've been motion, air quality, acceleration, or a host of other things. What matters is that I know how to get sensor readings, publish those readings to a database, then access that database to come up with front-end user interfaces.

THE FUTURE

There's room for expansion and improvement for this sensor suite. While it ended up as a climate sensor for a room, the original scope of the project was a room experience sensor which is why the noise sensor was there. However, the 4 data points weren't enough to be able to do that. Future iterations would introduce new data points: motion detection, time, perhaps even word triggers, and more. The outcome would be a system that allow the user to infer what's happening the room down to the most minute detail. Perhaps the user could tell that was a party in the house based on readings that show there was loud noise and a lot of movement at night. Or I could introduce particle sensors and the system could be used as a portable air quality index. Each new data point would require an exponentially robust enough GUI to accomodate everything, but it would be nice to see how far this project could be pushed.
